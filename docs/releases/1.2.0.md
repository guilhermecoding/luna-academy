# 1.2.0

Plataforma de gestão acadêmica — correções de chamadas, tema escuro em mobile e melhorias nas páginas administrativas.

## Destaques

- Horário de registro de chamadas corrigido com suporte a fuso horário
- Tema escuro em dispositivos móveis sem distorção de cores forçada pelo navegador
- Páginas administrativas com carregamento assíncrono e tipagem mais segura
- Integridade de presenças garantida por constraint de unicidade no banco

## O que mudou

### ✨ Novidades

- **Presenças:** constraint única `(lesson_id, student_id)`; remoção de duplicatas na migration; indicação de alunos desvinculados; busca de presenças considera `courseId`
- **Tema claro/escuro (mobile):** script bloqueante no `<head>`, `color-scheme: only light/dark`, componente `ThemeColorSchemeSync` e fallback CSS com `prefers-color-scheme`
- **Páginas admin:** migração para `PageProps`, redirecionamento em `/admin/[program]` e layouts refatorados

### 🐛 Correções

- **Chamadas:** `attendance_updated_at` migrado para `TIMESTAMPTZ(6)`; validação de timezone com fallback `America/Sao_Paulo`; `formatAttendanceUpdatedLabel` nas páginas de aula (admin e professor)
- **Carregamento admin:** `Suspense` + `PageSkeleton` em alunos, equipe, turmas, disciplinas e períodos; remoção de `loading.tsx` redundantes; `searchParams` integrados nas listagens
- **SAD:** correção na desestruturação de `useTransition` em `SADAccessTable`

### 🗄️ Banco de dados

| Migration | Descrição |
| --- | --- |
| `20260605120000_add_attendance_lesson_student_unique` | Remove presenças duplicadas e cria índice único `(lesson_id, student_id)` |
| `20260606043753_attendance_updated_at_timestamptz` | Converte `attendance_updated_at` para `TIMESTAMPTZ(6)` (dados existentes tratados como UTC) |

As migrations rodam automaticamente no deploy via `prisma migrate deploy`.

### ⚙️ Configuração

Nova variável opcional:

```env
NEXT_PUBLIC_APP_TIMEZONE=America/Sao_Paulo
```

Recomendado em produção para exibição correta dos horários de chamada.

## Deploy

1. Migrations executadas antes do build (script `build` já configurado)
2. Definir `NEXT_PUBLIC_APP_TIMEZONE=America/Sao_Paulo` em produção
3. Usuários mobile podem precisar limpar cache ou reabrir o PWA após o deploy
4. Registros antigos de `attendance_updated_at` gravados fora de UTC podem manter divergência histórica; novas chamadas ficam corretas
